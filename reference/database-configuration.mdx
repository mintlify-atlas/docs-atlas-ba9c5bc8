Kuzu is configured when you create a `Database` object. All settings are passed as constructor arguments — there is no separate configuration file. Once a database is opened, most settings cannot be changed without closing and reopening the database.

## Constructor parameters

<ParamField path="database_path" type="string | Path | None" default=":memory:">
  Path to the database directory on disk. If `None`, an empty string, or `":memory:"` is passed, Kuzu creates an in-memory database that is not persisted to disk and is destroyed when the process exits.
</ParamField>

<ParamField path="buffer_pool_size" type="int" default="~80% of system memory">
  Maximum size of the buffer pool in bytes. The buffer pool is the main in-memory cache for database pages. A larger buffer pool reduces disk I/O. Setting this to `0` uses the default (~80% of system memory).

  See [sizing the buffer pool](#sizing-the-buffer-pool) for guidance.
</ParamField>

<ParamField path="max_num_threads" type="int" default="CPU core count">
  Maximum number of threads Kuzu can use for query execution. Setting this to `0` uses the default, which is the number of logical CPU cores available. You can also adjust this per-connection after opening the database.
</ParamField>

<ParamField path="compression" type="bool" default="true">
  Enable on-disk compression for supported data types. Compression reduces storage size and can improve read performance when I/O is the bottleneck, at the cost of additional CPU usage during reads and writes.
</ParamField>

<ParamField path="lazy_init" type="bool" default="false">
  If `True`, the database is not initialized until the first query is executed. This is useful when creating the `Database` object in a parent process that will be forked, or when initialization should be deferred to a background thread.
</ParamField>

<ParamField path="read_only" type="bool" default="false">
  Open the database in read-only mode. In this mode, no write transactions are allowed. Multiple processes can open the same database path in read-only mode simultaneously. You cannot have a read-write database instance and a read-only instance open on the same path at the same time.
</ParamField>

<ParamField path="max_db_size" type="int" default="1 << 43 (8TB on 64-bit systems)">
  Maximum size of the database on disk in bytes. This controls the size of the memory-mapped address space reserved for the database files. On 32-bit systems the default is 1 GB. Adjust this downward in memory-constrained environments.

  <Warning>
    This parameter exists as a temporary workaround for mmap address space limits in certain environments and may be removed in a future release.
  </Warning>
</ParamField>

<ParamField path="auto_checkpoint" type="bool" default="true">
  If `True`, Kuzu automatically checkpoints the WAL to disk when the WAL file size exceeds `checkpoint_threshold`. Checkpointing merges WAL records into the main database files and resets the WAL.
</ParamField>

<ParamField path="checkpoint_threshold" type="int" default="16777216 (16MB)">
  WAL file size threshold in bytes that triggers an automatic checkpoint when `auto_checkpoint` is `True`. Setting this to `-1` uses the built-in default.
</ParamField>

<ParamField path="throw_on_wal_replay_failure" type="bool" default="true">
  If `True`, any failure during WAL replay at database startup raises an error. If `False`, Kuzu silently ignores the failure and replays up to the point where the error occurred. Set to `False` only if you need to recover a database with a partially corrupted WAL.
</ParamField>

<ParamField path="enable_checksums" type="bool" default="true">
  If `True`, Kuzu writes and verifies checksums for WAL records to detect corruption. Disabling this gives a small write performance improvement at the cost of reduced corruption detection.
</ParamField>

## In-memory vs. persistent databases

Kuzu operates in one of two modes depending on the `database_path` value.

<Tabs>
  <Tab title="In-memory">
    An in-memory database holds all data in RAM. It is not persisted to disk. Use in-memory mode for temporary workloads, testing, and data transformation pipelines where durability is not required.

    ```python
    import kuzu

    # Any of these create an in-memory database
    db = kuzu.Database()
    db = kuzu.Database(None)
    db = kuzu.Database(":memory:")
    ```

    <Note>
      In-memory databases do not support `spill_to_disk` or `read_only` mode. WAL and checkpoint settings have no effect on in-memory databases.
    </Note>
  </Tab>
  <Tab title="Persistent">
    A persistent database stores data in a directory on disk. If the directory does not exist, Kuzu creates it on first open.

    ```python
    import kuzu

    db = kuzu.Database("/path/to/mydb")
    conn = kuzu.Connection(db)
    conn.execute("CREATE NODE TABLE Person (id INT64, name STRING, PRIMARY KEY(id))")
    ```

    Data written in one session is available in subsequent sessions.
  </Tab>
</Tabs>

## Read-only mode

Opening a database in read-only mode allows multiple processes or instances to access the same database files concurrently without conflicting. This is the standard pattern for read-heavy workloads where writes are performed separately.

```python
import kuzu

# Multiple processes can open the same path in read-only mode
db = kuzu.Database("/path/to/mydb", read_only=True)
conn = kuzu.Connection(db)
result = conn.execute("MATCH (p:Person) RETURN p.name")
```

<Warning>
  You cannot have a read-write `Database` instance and a read-only `Database` instance open on the same path at the same time. Kuzu uses a file lock to enforce this constraint.
</Warning>

## Sizing the buffer pool

The buffer pool is the most impactful configuration parameter for query performance. It holds database pages in memory and reduces the number of disk reads required during query execution.

<Tip>
  For dedicated database servers, set `buffer_pool_size` to 60–80% of available system RAM. For shared environments, reduce it proportionally to avoid contention.
</Tip>

```python
import kuzu

# Set a 4 GB buffer pool
db = kuzu.Database("/path/to/mydb", buffer_pool_size=4 * 1024 ** 3)
```

If the buffer pool is too small relative to your working set, Kuzu must read data from disk on every query, significantly degrading performance.

## Thread configuration

Kuzu parallelizes query execution across threads. By default it uses all available CPU cores. You can reduce this at the database level or tune it per connection.

<Tabs>
  <Tab title="Python">
    ```python
    import kuzu

    # Limit the database to 4 threads globally
    db = kuzu.Database("/path/to/mydb", max_num_threads=4)

    # Override per connection
    conn = kuzu.Connection(db, num_threads=2)
    ```
  </Tab>
  <Tab title="Cypher">
    You can also set the thread count for a specific connection session using `CALL`:

    ```cypher
    CALL threads=4;
    ```
  </Tab>
</Tabs>

Reducing thread count is useful when running Kuzu alongside other CPU-intensive workloads, or when predictable single-connection latency matters more than parallel throughput.

## Compression

Compression is enabled by default for supported column types. It reduces on-disk storage and can improve read performance when I/O is the bottleneck on spinning disks or networked storage.

```python
import kuzu

# Disable compression — useful when CPU is the bottleneck
db = kuzu.Database("/path/to/mydb", compression=False)
```

<AccordionGroup>
  <Accordion title="When to disable compression">
    Consider disabling compression when:

    - You are running on fast NVMe storage where I/O is rarely the bottleneck
    - CPU resources are constrained and the encoding/decoding overhead is measurable
    - You are benchmarking and want to isolate raw storage performance

    Compression is a database-wide setting and cannot be changed without re-creating the database.
  </Accordion>
</AccordionGroup>

## WAL and checkpoint configuration

Kuzu uses a Write-Ahead Log (WAL) to ensure durability. Every write is appended to the WAL before being applied to the main database files. Checkpointing merges the WAL into the main files and resets it.

```python
import kuzu

db = kuzu.Database(
    "/path/to/mydb",
    auto_checkpoint=True,
    checkpoint_threshold=64 * 1024 * 1024,  # Checkpoint after 64MB of WAL
)
```

You can also trigger a manual checkpoint from Cypher:

```cypher
CHECKPOINT;
```

<Note>
  Kuzu forces a checkpoint when the database is closed cleanly (`force_checkpoint_on_close` is `True` by default). This keeps startup time fast by minimizing WAL replay.
</Note>

Setting `auto_checkpoint=False` defers checkpointing to manual `CHECKPOINT` calls, which gives you control over when the potentially expensive operation runs.

## Full configuration example

<Tabs>
  <Tab title="Python">
    ```python
    import kuzu

    db = kuzu.Database(
        database_path="/data/mydb",
        buffer_pool_size=8 * 1024 ** 3,   # 8 GB
        max_num_threads=16,
        compression=True,
        read_only=False,
        auto_checkpoint=True,
        checkpoint_threshold=32 * 1024 * 1024,  # 32 MB
        enable_checksums=True,
        throw_on_wal_replay_failure=True,
    )

    conn = kuzu.Connection(db)
    ```
  </Tab>
  <Tab title="Node.js">
    ```javascript
    const kuzu = require("kuzu");

    // Database(path, bufferManagerSize, enableCompression, readOnly, maxDBSize, autoCheckpoint, checkpointThreshold)
    const db = new kuzu.Database(
      "/data/mydb",
      8 * 1024 * 1024 * 1024,  // 8 GB buffer pool
      true,                     // enableCompression
      false,                    // readOnly
      0,                        // maxDBSize (0 = platform default)
      true,                     // autoCheckpoint
      32 * 1024 * 1024,         // checkpointThreshold (32 MB)
    );

    const conn = new kuzu.Connection(db);
    ```
  </Tab>
  <Tab title="Java">
    ```java
    import com.kuzudb.Database;
    import com.kuzudb.SystemConfig;

    SystemConfig config = new SystemConfig();
    config.bufferPoolSize = 8L * 1024 * 1024 * 1024; // 8 GB
    config.maxNumThreads = 16;
    config.enableCompression = true;
    config.readOnly = false;

    Database db = new Database("/data/mydb", config);
    ```
  </Tab>
</Tabs>

## Runtime settings

Some settings can be adjusted after the database is open using `CALL` statements on a connection. These affect the current connection session only.

| Setting | Type | Default | Description |
|---|---|---|---|
| `threads` | `UINT64` | CPU cores | Max threads for this connection |
| `timeout` | `UINT64` | `0` (disabled) | Query timeout in milliseconds |
| `progress_bar` | `BOOL` | `false` | Show query progress bar |
| `checkpoint_threshold` | `INT64` | `16777216` | WAL size threshold in bytes |
| `auto_checkpoint` | `BOOL` | `true` | Enable automatic checkpointing |
| `spill_to_disk` | `BOOL` | `true` | Allow spilling intermediate results to disk |
| `var_length_extend_max_depth` | `INT64` | `30` | Max depth for variable-length pattern expansion |

```cypher
CALL threads=8;
CALL timeout=30000;
CALL progress_bar=true;
```
